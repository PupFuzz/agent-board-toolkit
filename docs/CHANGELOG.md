## [Unreleased]

### Added
- **`tests/no-task-wrapper-selftest.sh` — a static regression guard against re-introducing the DL-219 `{task: {…}}` write-body wrapper** (card#4811). Follow-on hardening to card#4772: that cutover removed 8 wrapped sends, but two hid for two review rounds because they were hand-built escaped-quote string literals (`"{\"task\":{…}}"`) a plain `"task":` grep never saw. The new network-free selftest statically scans every regular file under `bin/` and fails if any **non-comment** line carries a `task`-keyed body wrapper in any of the three re-introduction shapes — unescaped JSON (`{"task":`), escaped literal (`{\"task\":`), or jq construction (`{task: {…}}` / `{task: $var}`) — via the matcher `("task"\s*:)|(\"task\"\s*:)|((^|[ ({,])task\s*:)`. The jq arm matches the wrapper key alone (not requiring a following `{`), so the brace-less argjson form is caught too; lines whose first non-whitespace char is `#` are excluded (the tree's one legitimate mention is the `# … {task: …}` doc note in `bin/kbcard`). A static test rather than a code helper because `bin/promote-released-cards` is standalone-vendored and cannot source the shared lib — and post-DL-219 there is no shared wrapper primitive left to centralize on — so only a source scan covers every bin uniformly. The test carries its own positive control (three temp fixtures, one per shape, asserted flagged — proving the matcher can fail) and a benign fixture (`task_id`/`to_task_id`/`from_task_id`/`task_links`/`.task-list`/`_action` + a `#`-comment mention, asserted **not** flagged). Added to the CI selftest matrix. No `bin/` or vendor-surface change.

## [0.20.0] - 2026-07-21

**Minor — the v3 task-write API cutover drops the `{task: {…}}` body wrapper (in lockstep with the kanban DL-219 prod deploy), plus a live-source archive gate and two `release-pr-body` syncs.** 5 PRs since v0.19.0 (#143–#147), cards #4772/#4747/#4751/#4761/#4762 (roundtable PupFuzz/agent-roundtable#39/#109/#111). The headline is the unconditional flat-body cutover (#147) — no shim, no version probe, so the tools move in lockstep with the kanban prod deploy carrying DL-219 — riding with `kbcard archive`'s new `may_archive` live-and-untwinned-source gate and the owner-side `release-pr-body` `tag_format` sync. Plus a docs-only solo-orientation sync (coord v0.10.0). `promote/action.yml` vendor surface unchanged.

### Added
- **#146** — **`kbcard archive` gates on the shipped `may_archive` invariant, with an audited `--force` override** (card#4747, roundtable PupFuzz/agent-roundtable#39). Before the archive `PATCH`, `kbcard archive` now consults the framework's shipped `kanban_common.may_archive` primitive (path-loaded version-agnostically, the same ladder `board-session-close` uses for the reconcile hook) via a thin new `bin/_kbc-may-archive.py` shim: it builds a tri-state per-source resolver over the card's backing coordination sources (`pr_number`/`issue_number` by-ref → `gh pr|issue view --json state`, mapping `OPEN`→live, `CLOSED`/`MERGED`→terminal, a read that 404s/errors→unresolvable; an `id:<sid>` provenance tag → unresolvable, since kbcard has no GitHub handle for a coordination stable-id) and passes the board's live cards as `surviving_cards` for twin-awareness. If a backing source is still **live** (or unresolvable — fail-closed) and no surviving twin card keeps it discoverable, the archive is **refused** (rc 1, no `PATCH`) with a drift-vocabulary line (`⚠ archive withheld: card #<id> — <reason>`) — archiving the only card of a live source would hide it from the reconcile/detect backstop. `--force` performs the archive anyway and records an **audited** override (`⚠ archive FORCED past live-source gate: …` to stderr + the kbcard failure log), never a silent bypass. If the `may_archive` primitive cannot be located the gate **fails loud** (refuses, names `--force`) rather than archiving unchecked. A card fetch that returns HTTP 2xx with `.data` **null/absent** (a trashed / permission-limited / edge card) is treated as fail-closed at **both** choke points — the bash `_kbc_archive_decision` short-circuits to `noprimitive` before the board fetch, and the shim rejects an explicit-null `card` — so a null card can never collapse to a source-less `{}` that would false-`ok` and archive a live-and-untwinned card (the card-side sibling of the board input's safe `[]` default). And a **gate-infra failure never blocks `--force`**: if the decision can't be computed (e.g. `python3` absent), `--force` still archives AND still writes the audited override line (`… infra could not run — forced`), so the escape hatch isn't held hostage by the gate it exists to bypass — while the non-force path stays fail-closed (a gate-infra failure refuses). The by-ref GitHub-state read reuses the framework's own `gh_json`/`gh` credential path; the board→repo fallback reads `kanban.boards[]` from the coord config, and a card's own `pr_url`/`issue_url` source is derived via the framework normalizer when a descriptor carries no repo. `--dry-run` is unchanged (offline, no gate evaluation, no mutation). New `tests/kbc-may-archive-selftest.sh` exercises the shim's resolver tri-state over the **real** primitive with a fake `gh` (skipping the primitive-backed cases only where the plugin's `kanban_common` is absent), and `tests/kbcard-selftest.sh` gains the wiring cases (refuse / forced-audit / ok / fail-loud), red-when-reverted (removing the gate lets a blocked card archive). `promote/action.yml` vendor surface unchanged.

### Changed
- **#147** — **Task write bodies are sent FLAT (top-level), dropping the `{task: {…}}` wrapper** (card#4772). The kanban v3 write API unified to take task fields at the top level of the request body and no longer accepts the `{task: {…}}` envelope (kanban DL-219); every task `POST`/`PATCH` these tools issue now sends the task object flat. This is an **unconditional** cutover — no version probe, no wrapped fallback — so the tools must run against a kanban deploy that carries DL-219 (there is no shim; they move in lockstep with the kanban prod deploy). Migrated (8 task-write sites across 5 files, found by enumerating every `/tasks` write rather than grepping for `task:` — two were hand-built escaped-quote literals a name grep slid past): `kbcard` `create-card`/`move`/`patch`; the DL-stamp write **and** the move-to-In-Progress write in `board-card-start`; the tag backfill in `dl-a0-backfill-triaged`; the throwaway-card create + dl-clear in `dl-a1-register-field`; and the stage-move in `promote-released-cards`. Unaffected: `task_links` bodies (already flat, different endpoint) and `_action` bodies (never wrapped). `tests/kbcard-selftest.sh` now asserts the flat body shape (create-card tags/type, the shared create/patch payload, the swimlane setter). `promote/action.yml` vendor surface unchanged.
- **#145** — **`board-session-close` delegates its inverse-drift leg to the shipped `kanban-reconcile.py --detect`** (card#4751, roundtable PupFuzz/agent-roundtable#109 + #111). The bespoke bash inverse-drift check (In-Review cards whose recorded `pr_number` matches no open PR) is retired in favor of the framework-maintained detect hook — the sanctioned conforming implementation of the inverse-drift CONTRACT (the framework's `BOARD-SYNC.md` § "The detect backstop contract"), which previously named this bash as an accepted-divergence conforming implementation "until the solo hook's adoption completes." Both implementations were proven equivalent before the swap: on a synthetic positive control they flag the same two drift sub-cases (stale ref + missing correlation) and suppress the same clean card; on the live boards both report no inverse drift. The delegated hook is a net gain — it adds a **forward** leg (an open PR of a board's repo with no live card carrying its ref) the bash never had, and its inverse leg is repo-qualified `(source, ref)` tuples rather than a flat ref-number set. The hook path is resolved **version-agnostically** (explicit `$KANBAN_RECONCILE_HOOK` override → the coord plugin dir on `$PATH` → the marketplace clone → the newest cached version by `sort -V`) so a plugin update never breaks it, and an unresolvable hook **fails loud** (rc 1, "DID NOT RUN") rather than silently skipping the drift check; a non-zero hook exit propagates so a config/API failure isn't read as a reconciled board. New `tests/board-session-close-selftest.sh` (network-free) covers the resolver ladder and the surfacing / fail-loud / rc-propagation paths (verified red-when-reverted: 3 mutations each red their guarding check). Because a board without `inverse_check_columns` runs the forward leg only and reports `inverse check OFF` at rc 0 — a silent reduction of coverage where `board-session-close` previously keyed success purely on the hook's exit code — the ritual now scans the hook output and **re-emits the OFF state as a loud `⚠` warning on stderr**, naming the affected board key, outside the per-board `(no drift detected)` framing, so the degraded (inverse-off) coverage is never read as a clean pass. A board wholly `skipped` by the hook (absent from `kanban.boards[]`, or lacking a single-repo `repo`) is surfaced the same way. OFF is **not** a hard error (a board may intentionally not arm inverse) — the exit code and forward-leg output are unchanged; only the warning is added. **Adoption note:** the inverse leg is armed by an `inverse_check_columns` entry per board in `coordination.config.json` (`kanban.boards[]`); without it the hook runs the forward leg only and `board-session-close` warns that the inverse-drift check is off for that board.
- **#143** — **`release-pr-body` `tag_format` config knob + 2–4-segment version extraction** (card#4761, roundtable PupFuzz/agent-roundtable#39). Adopted from the agent-board-framework's `templates/release/release-pr-body`, which had evolved ahead of the toolkit copy: a new optional `.tag_format` key in `.release-pr.json` (default `v{{version}}`, preserving the historical behavior byte-for-byte) maps a version to its git tag name for non-`v`-prefixed schemes (`{{version}}`, `release-{{version}}`, …) — and the own-tag exclude in the baseline resolution now derives from it instead of a hardcoded `v` prefix, so a re-run after tagging under a non-`v` scheme no longer reports the current release as its own previous tag. Version-file extraction accepts 2–4 numeric segments (`[0-9]+(\.[0-9]+){1,3}`), so a .NET `Major.Minor.Build.Revision` version (`1.22.1.0`) is no longer silently truncated to `1.22.1`. Both behaviors gain red-when-reverted selftest coverage (verified: 5 checks fail on the pre-sync bin). **Ownership declared** (README): the toolkit owns both release bins (`release-pr-body` + `promote-released-cards`) — tests and release discipline live here; the framework's `templates/release/` copies are mirrors synced at toolkit tags, closing the fork this change reconciles.

### Fixed
- **#144** — **`release-pr-body` display header renders the real tag via `tag_format`** (card#4762). Owner-side follow-on to the card#4761 sync: the PR-body header hardcoded `v${VERSION}` independent of `tag_format`, so a non-`v` scheme showed a `v`-prefixed header while the actual tag differed. The header now renders the tag derived from `tag_format`, and the tag computation is hoisted ahead of the baseline block so it is also live on the explicit `--base` path; under the default `v{{version}}` the output is byte-identical. Red-when-reverted selftest coverage (verified: 3 checks fail on the pre-change bin, including the `--base` path).

## [0.19.0] - 2026-07-19

**Minor — the swimlane write-side lands: `kbcard patch --swimlane` plus a defined-swimlane enumeration on every unresolved name.** 1 PR since v0.18.0 (#140, bundling cards #4714 + #4713, roundtable PupFuzz/agent-roundtable#101 + #100), plus a docs-only solo-orientation sync. Both features activate only on a board env that declares `KB_SWIMLANE_<id>=<name>` entries; `promote/action.yml` vendor surface unchanged.

### Added
- **#140** — **`kbcard patch --swimlane <name|id>` — write-side swimlane setter** (card#4714, roundtable PupFuzz/agent-roundtable#101). Closes the read/write asymmetry left by `list --swimlane` (the read filter): a card could be listed by swimlane but only assigned to one via a raw `swimlane_id` card `PATCH`. The new flag writes the card's top-level `swimlane_id` column (rides the inner task object like `--column`'s `workflow_stage_id`, not the payload merge), resolving a name through the **same** `swimlane_id()` helper `list` already uses — no second resolver. `--swimlane none` (or `--swimlane 0`) is the explicit un-assign form, writing `swimlane_id` null; every other ref resolves name→id (numeric ids pass through) and fails loud (rc 2) on a typo. The write echo surfaces the resulting `swimlane_id` only when `--swimlane` was set (parity with it always showing `workflow_stage_id`), so other patches keep their echo shape. Additive and `patch`-only: `move` stays column-only.

### Fixed
- **#140** — **An unresolved `--swimlane <name>` now enumerates the board's defined swimlanes** (card#4713, roundtable PupFuzz/agent-roundtable#100). `swimlane_id()` printed `swimlane 'X' is not defined on this board` and stopped — with nothing listed, a reader reasonably infers the board has *no* such swimlane (e.g. missing a required lane prefix), and could report that false board fact. The error now appends the id→name map `_kbc_swimlane_map()` already builds (`defined swimlanes: 1=…, 2=…`), mirroring the board-not-found error's "boards found on this box: …" style; a board with no swimlanes declared gets a distinct honest line rather than an empty list. Error text only — the rc-2 loud-fail contract is unchanged, so `list --swimlane` still exits 2 on a typo.

## [0.18.0] - 2026-07-19

**Minor — one new `kbcard` correlation surface plus a fail-loud config error, riding on three same-behavior consolidations.** 5 PRs since v0.17.0 (#133–#137). The feature is issue↔card correlation stamping (`--issue`/`--issue-url`); the fix makes an unconfigured-board error name its own remedy; the remaining three are duplication-removal refactors with byte-identical runtime behavior. No config migration; `promote/action.yml` vendor surface unchanged.

### Added
- **#136** — **`kbcard --issue`/`--issue-url` for issue↔card correlation.** New optional flags on `create-card` and `patch` stamp `payload.issue_number` (a JSON number, matching `pr_number`'s numeric canonicalization) and `payload.issue_url` (which drives the by-ref `source` owner/repo for multi-repo boards, exactly as `pr_url` does), so a card can be correlated to a tracked issue without dropping to a raw per-key payload `PATCH`. The two keys are independent and co-stampable — a card can carry both `issue_*` and `pr_*` with no exclusivity check, the way `dl`/`pr` already coexist. `adopt-to-dl` gains a matching optional `--issue <N>` co-adoption path that co-stamps `issue_number` + a source-qualified `issue_url` in the same atomic write and fail-loud-verifies the `github_issue` by-ref parallel to the DL one. Additive only: omitted flags leave the keys absent (not null), nothing previously accepted is now rejected, and adoption without `--issue` is byte-identical to before. The shared `_kbc_build_payload` helper is extended to carry the new keys, so `issue_number` inherits the same numeric coercion `pr_number` already had rather than a duplicated merge.

### Fixed
- **#133** — **the unconfigured-board error now names its fix, and default-board resolution is documented.** On a box whose primary board env file isn't named `dev`, a bare invocation failed with `board env file not readable: …` and no path forward, reading as broken when it was merely unconfigured for a default. The `rc=2` board-env arm — the one sibling in `kb_load_config` that printed no remedy — now names `--board <name>` / `KBCARD_BOARD_ENV` (context-aware for the bare-default case vs a named board whose file is absent) and lists the `~/.kanban-*-board.env` files that do exist on the box. `INSTALL.md` documents `KBCARD_BOARD_ENV` as the flag-free option and adds the error to troubleshooting. Docs + error text only — the rc, the trigger, and default-resolution behavior are unchanged. Covers every `kb_load_config` consumer (`kbcard`, `adopt-to-dl`, `dl-a0-backfill-triaged`, `dl-a1-register-field`).

### Changed
- **#134** — **`_kbc_build_payload` shared by `create-card` + `patch`.** Four verbatim-duplicated payload-assembly blocks in `bin/kbcard` (the args→JSON merge `jq`, the `version_target` board-capability guard, the DL-canon append, and the `pr_number`/`pr_url` appends) are hoisted into one helper both commands call. Pure same-behavior refactor: the assembled `task.payload` is byte-identical to what each command produced before — numeric-string→JSON-number coercion, the version guard and its warning, and DL canonicalization all preserved; no command's accepted/rejected inputs or emitted payload change.
- **#135** — **selftest harnesses consolidated onto a shared `tests/_selftest-prelude.sh`.** The harness every `tests/*-selftest.sh` duplicated verbatim (`ok`/`bad`/`fails`, `eq`, `expect_rc`/`expect_out`, the required-bin guard, the scratch-dir setup, and the trailing PASS/FAIL summary) moves to one sourced prelude, removing nine copies and normalizing summary wording that had drifted across files. Harness-only: no test case or assertion changed, and all selftests pass identically before and after.
- **#137** — **assorted intra-file + CI deduplication (dedup-audit finding D6), same-behavior throughout.** Single-sources several rules that were each written twice, executable behavior byte-for-byte preserved: a `max_int` "highest int in a stream" helper shared by `next-dl`'s two scans (with new `tests/next-dl-selftest.sh` coverage it previously lacked); the throwaway-teardown request bodies in `dl-a1-register-field`; a shared jq leading-zero normalization across `promote-released-cards`' `dl_number`/`pr_number` fields; and a doubled token-file ladder in `board-snapshot`. On the CI side, `auto-tag-version.yml` now reads the version shape from `.release-pr.json`'s `version_regex` instead of a divergently-anchored hardcoded copy, and the byte-identical per-selftest jobs collapse to a `strategy.matrix` (no required check renamed; `shellcheck`, `drift-check-selftest`, and the composite-action smoke test stay separate). `promote/action.yml` / `_kb-board-lib.sh` unchanged — vendored consumers see only an internal refactor.

## [0.17.0] - 2026-07-18

**Minor — an 8-PR quality release: two roundtable-driven `kbcard`/hook features, a pagination-completeness fix ported into both board movers, a DL-primitive consolidation, and an error-body-preserving `board-card-start` refactor.** 8 PRs since v0.16.0 (#123–#130). Cards #4510/#4513/#4514/#4413/#4617/#4621/#4623/#4637. Two new feature surfaces (`create-card --triaged`, `kbcard list --swimlane`); no config migration; `promote/action.yml` vendor surface unchanged.

### Added
- **#124** — **`kbcard create-card --triaged` flag** (card #4617, roundtable FR PupFuzz/agent-roundtable#86). Mirrors the existing `patch --triaged` onto `create-card` so a card can be born triaged — a plain PM-minted product card that forgot `--tags triaged` otherwise re-surfaced as a false-positive untriaged card every session (the untriaged check keys on a missing `triaged` tag). On a fresh card the tags CSV is built from scratch, so a plain append suffices (no read-merge-write like the `patch` path). Scope is the generic `--triaged` verb only; any PM policy tag stays caller-passed. Works on native-type, tag-typing, and mixed boards.
- **#125** — **`board-card-start` accepts the glued `card<N>` branch spelling + a fail-soft pre-push branch-name lint** (card #4621). The post-checkout auto-move required a separator after `card`, so the natural glued spelling `fix/card4524-slug` matched no grammar — the hook fired, found no token, and the card silently never moved to In Progress. The separator is now optional (`card[-/#]?`), so `card4524`/`card-4524`/`card/4524`/`card#4524`/`#4524` all correlate (still requires the literal `card`/`#` at a token boundary + ≥2 digits, so DL tokens and version numbers stay safe). A new fail-soft `pre-push` lint warns (never blocks; always exit 0) only on a card-ish token the grammar still won't accept (e.g. `card_4524`), reusing the exact matcher the mover uses so the two can't disagree. The widened grammar affects the local hook only; the bridge writeback still requires `card-<id>`/`card#<id>`, which stays the canonical full-lifecycle convention.
- **#130** — **`kbcard list` surfaces `swimlane_id` + a resolved name, and a `--swimlane <name|id>` filter** (card #4637, roundtable FR PupFuzz/agent-roundtable#88). Per-swimlane triage previously dropped to raw-API glue because `list` omitted swimlane. Names resolve from the board env, id-keyed `KB_SWIMLANE_<id>=<name>` (the inverse of `KB_STAGE_<name>=<id>`, since swimlane names are freeform and can't be env suffixes) — zero extra API call. `--swimlane` composes with `--column`/`--type`; a typo'd name errors loud (rc 2), a numeric id passes through. The filter keys on the stringified id because `swimlane_id`'s API type is unverifiable on any reachable board (no board has swimlanes), where a numeric `==` would silently drop a string-typed id.

### Fixed
- **#127** — **`promote-released-cards` now censuses the whole-board read against `meta.total` so a short read can't silently promote a partial board** (card #4513). The mover's inline pagination had drifted from the lib's `fetch_board_cards` — it broke solely on a <200-row page and never consulted `meta.total`, so a server short read terminated the scan early and promoted from an INCOMPLETE board (a card its scan misses is a card it leaves un-promoted). Extracted into a self-contained `fetch_whole_board()` (the standalone must not source the lib) that captures `meta.total`, tracks a pre-dedup running sum, and after dedup distinguishes a genuine undercount (sum < total → DIE, matching the tool's page-cap refuse policy) from a page-boundary dedup artifact (sum ≥ total → warn only).
- **#129** — **`fetch_board_cards` defaults `meta.last_page` to UNKNOWN, not `1`** (card #4623). The lib defaulted `last_page` to `1` and broke unconditionally on `page >= last_page`; a server returning a full 200-row page 1 with neither `meta.last_page` nor `meta.total` present would skip the census and break at page 1 — a silently truncated read. Unreachable against the real Laravel `LengthAwarePaginator` (always emits both), hence latent, but it is the exact miss #4513 already guarded in the co-vendored standalone. Ported for parity: `last_page` is usable only as a positive integer (absent/non-numeric/<1 → unknown), and the secondary break fires only when the server positively declares it — byte-identical behavior on the real API path; a meta-absent read only becomes MORE complete.

### Changed
- **#123** — **`board-card-start` routes its card read + two writes through `kb_api`/`kb_api_status`** (card #4510). The read used a local `get()` and the dl-stamp + move-to-In-Progress writes used `curl -fsS`, which discards the HTTP status and the 4xx/5xx body — exactly the 403-scope-vs-422 indistinguishability the shared lib was fixed to remove (#4337), a fix that never reached these local calls. The "kept local so it sends THIS board's token" rationale was void (`kb_api` reads `KB_TOKEN`, which this hook already set). The two writes now go through a shared `_bcs_patch` helper that captures `KB_HTTP` and durably records a failure via `kb_bcs_log` — the previously-silent failed stamp/move now leaves a trace. Success-path behavior unchanged.
- **#126** — **DL int/format/by-ref-hit consolidated into shared `_kb-board-lib.sh` primitives** (card #4514, dedup-audit finding D5). Three DL-family duplications hoisted: `kb_dl_int_lenient` (the lenient client mirror of the server's `canonicalize('dl')` key, migrated from `adopt-to-dl`, kept a deliberate sibling of the strict `kb_dl_num`), `kb_by_ref_hit` (the "contains card N" predicate centralized from two forks; the lib keeps the object-or-array-tolerant form, so `adopt-to-dl`'s verify safely widens), and one canonical `DL-NNNN` formatter (`kb_dl_canon`) replacing six inline `DL-%04d` literals — `next-dl` gains a fail-loud guard on a computed DL outside the canonical range. `promote/action.yml` consumers unaffected.

### Docs
- **#128** — **`ADOPTION.md`'s "Is this for you?" gate keys on board API access, not instance ownership** (card #4413). The gate read adoption onto instance ownership ("your own kanban board (your own instance)") when the toolkit's real precondition is board API access (an api base + token + board id; it never reads who operates the host) — the parenthetical read successful tenant adopters out of the audience in the doc's first decision paragraph. Reworded to board access, naming the tenant case explicitly; consumer-agnostic.

## [0.16.0] - 2026-07-17

**Minor — three roundtable-driven board-tooling deliverables: `kbcard` card-removal verbs + two `board-snapshot` fixes.** 3 PRs since v0.15.1 (#118–#120). Cards #4479/#4448/#4447. One new command surface (`kbcard archive`/`delete`); the two `board-snapshot` changes are a correctness fix + a perf fix with no config migration.

### Added
- **#118** — **`kbcard archive` + `kbcard delete [--hard]` card-removal verbs** (card #4479, roundtable FR #74 sola-pm). The board API has no HTTP `DELETE` — removal is a non-discoverable top-level PATCH `_action` plus a two-step force-delete — so agents were hand-rolling `curl` to remove a test card. `archive` (`_action:archive`) takes a card off-board as a recoverable live row (keeps its id + `dl_number`); `delete` (`_action:delete`) trashes it recoverably within the retention window (**retains** `dl_number`); `delete --hard` soft-deletes then `POST force-delete.json` for a permanent cascade (**releases** `dl_number`). Thin orchestration over the existing `resolve_task` + `kb_api` primitives (mirrors `cmd_move`/`cmd_patch` — no write-path reimplementation); `--dry-run` + a `--yes` confirmation gate on the destructive paths.

### Fixed
- **#119** — **`board-snapshot`'s `untriaged()` inherited a sibling board's stage ids** (card #4448). `board-snapshot` reads each board's config by **sourcing** its env file (which `export`s `KB_BOARD_ID` + the `KB_STAGE_*` ids); `snap()` scrubbed a hardcoded subset before the next board but `untriaged()` unset only `KBCARD_TOKEN_FILE`, so in an operator shell that had sourced board A, a board B env **omitting** any terminal-stage id (`KB_STAGE_SHIPPED_TO_DEV`/`RELEASED_TO_MAIN`/`WONT_DO`) or `KB_BOARD_ID` silently inherited A's value — B's terminal set went wrong and cards were mis-suppressed or wrongly flagged UNTRIAGED, a **quiet** failure of the "triage is never silently missed" contract. `untriaged()` now scrubs the same full set `snap()` does and gains the `KB_BOARD_ID` guard.

### Changed

## [0.14.0] - 2026-07-15

**Minor — a security fix in the `api_base` host guard, plus the restoration of the board-env token precedence that v0.8.2 silently broke.** 2 PRs since v0.13.0 (#98, #97). Cards #4346/#4325.

**Read before upgrading — two consumer-visible changes.** (1) **[vendor]** a repo that vendored `promote-released-cards` **must re-vendor** to get the security fix; the vendored copy carries its own guard and does not track the toolkit. **[release-CI]** SHA-pinned action consumers get it with the pin bump (`promote/action.yml` is unchanged). (2) **[host]** if you set `KBCARD_TOKEN_FILE` in **both** a board env and `~/.kanban-host.env`, the **board's** token now wins where the host's silently did since v0.8.2 — a live change in *which credential is sent*. Full actions: `docs/UPGRADE.md` § v0.14.0.

### Security
- **SECURITY — the `api_base` host guard accepted a URL that sends the bearer token to an attacker-controlled host.** `kb_require_https_host` (`_kb-board-lib.sh`) and its standalone mirror `host_ok` (`promote-released-cards`) terminated the URL authority at **`/` alone**. RFC 3986 ends it at the first of **`/`, `?`, or `#`** — so with a delimiter placed before an `@`, the userinfo strip (`${h##*@}`) reached past it and lifted a fake host out of the query/fragment:

  ```
  api_base: https://evil.example#@kanban.your-host.tld
  guard parses host -> kanban.your-host.tld  -> ACCEPTED
  curl connects to  -> evil.example          -> bearer token sent to the attacker
  ```

  Because `api_base` is read from the **committed, PR-editable** `.release-pr.json`, a pull request alone was enough: for `board-card-start` a maintainer merely checking out the branch leaked their kanban token, and for `promote-released-cards` a repo running promote on untrusted config would leak the `KANBAN_WRITEBACK_TOKEN` CI secret. Neither required any access to `$HOME` or `git config`. Both copies now terminate the authority at `[/?#]`. Verified end-to-end against a listener standing in for the attacker: the pre-fix hook delivered the bearer token, the fixed hook sends nothing and names the real host in its refusal. Legitimate values — subdomains, `:port`, real `user:pw@` userinfo, queries — are unaffected.

  It survived because the guard had **no decision coverage**: it was wired and ran on every checkout, but nothing had ever asked it to judge a hostile URL. New `tests/kb-host-guard-selftest.sh` (+ CI job) pins the matrix and additionally asserts the two copies agree — they are sync-paired by comment only, and both carried the identical defect.

  **[vendor] Action required:** `promote-released-cards` changed, so a repo that **vendored** it must re-vendor (UPGRADE §3) to get the fix. **[release-CI]** consumers of the SHA-pinned composite action get it with the pin bump — `promote/action.yml` itself is unchanged.
### Fixed
- **A board env's `KBCARD_TOKEN_FILE` is honored again — the documented precedence is restored.** `kbcard` honored it from v0.4.2 through v0.8.1 (`v0.8.1:bin/kbcard:440`, *"after host + board env are sourced, so a config-file `KBCARD_TOKEN_FILE` is honored"*). The **v0.8.2 shared-library extraction** consolidated six divergent copies of the config resolution onto the two that had the source order backwards (`next-dl`, `dl-a1-register-field`) and **silently regressed** it: from v0.8.2 through v0.13.0 a board-env `KBCARD_TOKEN_FILE` was ignored and the host/default token was sent to every board. The docs were right; the code drifted. One ladder now serves every tool — **board > host > ambient > `~/.kanban-dev-token`** — as a property of source order (host sourced first, board second) rather than six hand-rolled precedence tests. **Operator-visible where a board env *and* the host env both set `KBCARD_TOKEN_FILE`:** the board's token now wins, changing which credential is sent — see the `docs/UPGRADE.md` v0.14.0 entry.
- **A board env that sets the board-independent `KBCARD_API` is refused, loudly.** Same inverted order, same silence: v0.4.2–v0.8.1 ignored one (as INSTALL.md documented, because `API` was frozen before the board env was sourced), while v0.8.2–v0.13.0 sourced the board env *first* and so silently **honored** it — quietly re-pointing the tool at another host. `kb_resolve_env` now returns rc 4 and names the offending file; `next-dl` no longer swallows that message (it muted stderr, leaving a bare "config incomplete" while losing both the atomic DL claim and the board's `dl_number` seed — the DL-154 re-mint collision class).
- **`board-snapshot` no longer goes dark on a host with no `~/.kanban-dev-token`.** It read one token up-front and `exit 0`'d when that file was missing, so a host using only per-board tokens rendered **nothing**. It now reads each board's token as it renders that board, honoring that board's `KBCARD_TOKEN_FILE`; a board whose token is unreadable reports itself instead of blanking the entire snapshot.
- **`board-card-start` attributes a failed card read to its actual cause.** `get()` used `curl -f`, so an unreachable API, a rejected token (401/403), and a server error were all an empty body — every one reported as **"card … does not exist"**, sending an agent to hunt a missing card instead of a broken token. Each cause is now distinct in the durable diagnostic log. (The status is returned on stdout: `$(get …)` is a subshell, so a global set inside it cannot reach the caller.)

### Changed
- **`board-card-start`: per-board tokens are a host-local opt-in, and `git config kanban.board-id` outranks `.release-pr.json`.** The hook consults a board env's `KBCARD_TOKEN_FILE` **only** when the repo's board id came from the repo-local `git config`; a board id from the committed, PR-editable `.release-pr.json` keeps the host/default token (identical to prior behavior). Reordering the two sources alone would not have closed this — they are documented mutually-exclusive populations, so the inversion only helps the population that never had the exposure; gating the *capability* on the trusted source is the actual fix. The committed value is now digits-validated (`^[0-9]+$`) before it keys a board-env lookup.
- **Reading a board env is now clean-slate** (`kb_board_env_get`): every key is unset before the file is sourced, so a key the file does not set reads as **empty** instead of inheriting whatever is in the environment. Board envs `export` their keys, so in an operator shell that sourced one board's env, that board's values are live for every later tool. Scope, precisely: stage ids are unique to one board, so an inherited id never matches another board's cards — the reachable cases are `board-snapshot` (a board env with no `KB_BOARD_ID` would render a *different* board's cards under this board's label, or abort the subshell under `set -u`) and a board env missing a required stage id (the guard would pass on an inherited value and PATCH a stage that isn't on that board). `board-snapshot` now also says so (`• <label>: (env … sets no KB_BOARD_ID)`) rather than silently rendering nothing.
- **`board-card-start` resolves its board env once**, using it for both the stage ids and the token — the two lookups were separate matchers, free to disagree about which file wins when two board envs claim the same `KB_BOARD_ID`. A duplicate `KB_BOARD_ID` now warns (which file wins is arbitrary either way; being silent about it was the defect), and the stage ids are read newline-delimited so an unset optional `KB_STAGE_HELD` cannot shift another id into its place.
- **`kb_load_host_token`'s `gated`/`unconditional` mode flag is gone**, replaced by `kb_load_host_env`. The two modes differed only because the gate conflated "source the host env" with "let the host's `KBCARD_API` win"; separating those makes one behavior serve both callers. It also fixes the gated bug where a stray ambient `KBCARD_API` skipped the host env entirely, dropping the host's `KBCARD_TOKEN_FILE` — `board-snapshot` was the only `gated` caller, so it was the only tool affected.

### Added
- **`tests/kb-board-lib-selftest.sh` + a CI job** — network-free unit checks for the shared config resolution under a scratch `HOME`: the full token ladder in all four cases, the `KBCARD_API` refusal and its message, the failure return codes, the board-env matcher (no-match, a leaked `KB_BOARD_ID` false-match, duplicates, an unparsable env), and token paths containing spaces. The ladder is a property of source *order*, which is precisely how it regressed unnoticed for five minor versions — nothing exercised it.
